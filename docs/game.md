# 僕が考えた最強のエンジニアたち！！

## 概要

GitHubのあなたをソシャゲに！  
仲間を集め、大会で勝利をつかもう！

**キーワード**: シングルプレイ, マルチプレイ（非同期対人）, ソシャゲ

---

## ゲームの基本フロー

1. GitHubアカウントをOAuth連携し、自分のキャラクターを生成する
2. フレンドを追加し、デッキメンバーを集める
3. ハッカソン大会にエントリーし、デッキスコアが近い相手のデッキとシミュレーション対戦して開発スコアを競う
4. GitHubアクティビティの更新でキャラクターを成長させる

---

## GitHub OAuth 連携

### 認証方式

GitHub OAuth App または GitHub App を使用してユーザー認証を行う。  
ユーザーは初回ログイン時にGitHubアカウントとの連携を許可する。

### 必要なスコープ（パーミッション）

| スコープ    | 用途                                                   |
| ----------- | ------------------------------------------------------ |
| `read:user` | プロフィール情報（アバター・名前・bio）の取得          |
| `repo`      | **パブリック・プライベート両方**のリポジトリ情報の取得 |

> 現在の実装では OAuth 認証時に `read:user` と `repo` を利用する。`read:org` は未使用。

> **プライバシーへの配慮**: 取得したデータはスキル値の算出にのみ使用し、リポジトリの具体的な内容（コード・コミットメッセージ等）は保持しない。スキル計算後は生データを破棄する。

### 取得データの範囲

OAuth連携後、GitHub APIを通じて以下のデータを取得する。

| データ種別                 | 対象                                 | 用途                               |
| -------------------------- | ------------------------------------ | ---------------------------------- |
| リポジトリ一覧             | パブリック＋**プライベート**（全件） | 保有技術の集計、企画力スコアの算出 |
| コミット履歴               | 過去1年分（全リポジトリ）            | 開発速度スコアの算出               |
| README / wiki / Discussion | 全リポジトリ                         | 企画力スコアの算出                 |
| Pull Request 履歴          | 過去1年分                            | 開発速度ボーナスの算出             |
| プログラミング言語統計     | 全リポジトリ（Byte数）               | 主要技術・保有技術の判定           |
| コントリビューション草     | 過去1年分                            | スキル傾向の判定補助               |

> **注意**: Forkリポジトリは算出対象から除外する（自分が実際に開発したものではないため）。  
> ただし、Forkしたリポジトリへの自己コミットが存在する場合はコミット数のみカウント対象とする。

## キャラクター

### 生成方法

GitHub OAuth連携後、GitHub APIで取得したデータをもとにキャラクターを自動生成する。  
パブリック・プライベートを問わず**全リポジトリ**を解析対象とするため、普段から非公開で活動しているエンジニアのキャラクターもより正確に反映される。

> 現在の実装では GitHub の `avatar_url` をそのままキャラクター画像として利用する。任意アイコンアップロードは未実装。

### レベルアップ

| トリガー                 | 内容                                                                                     |
| ------------------------ | ---------------------------------------------------------------------------------------- |
| GitHubデータの再スキャン | ゲーム内で「GitHubを再スキャン」を実行すると、最新データをもとに各スキル値が再計算される |

> **注意**: スキル値が下がることもある（リポジトリ活動が減少した場合など）。現実のエンジニア活動がそのままゲームに反映される。
> 現在の実装ではレベル制は採用しておらず、成長はスキルスコア・デッキスコア・ランク表示で表現している。

### フレンドシステム

「フレンドに追加する」機能でフレンドのGitHubユーザー名を入力すると、そのエンジニアのデータからキャラクターが自動生成される。  
生成されたキャラクターは自分のデッキに組み込むことができる。

- フレンドのキャラクターはフレンド本人がゲームに登録・再スキャンを行ったタイミングで更新される
- フレンド未登録のGitHubアカウントからキャラクターを生成する場合、**パブリックリポジトリのみ**を参照してキャラクターを生成する
- フレンドがゲームにログイン・OAuth連携済みの場合、本人の同意のもとでプライベートデータも反映されたキャラクターが使用される

> **設計意図**: プライベートリポジトリは本人のみが許可・管理できるデータであるため、フレンドキャラクターに反映できるのは本人がゲームに登録・連携した場合に限る。

#### 承認制オプション

ユーザーは設定から「フレンド追加を承認制にする」を有効化できる。

| 設定                          | 挙動                                                                                       |
| ----------------------------- | ------------------------------------------------------------------------------------------ |
| 承認制 **オフ**（デフォルト） | フレンド申請を受けると即時キャラクター生成を許可する                                       |
| 承認制 **オン**               | 申請通知が届き、承認した相手にのみキャラクター生成を許可する。拒否・無視した場合は生成不可 |

> 承認制をオンにしていても、自分からフレンド申請を送った相手については相手側の設定に関わらずキャラクター生成が可能。

## スキル

### スキルの種類（上限なしスコア）

すべてのスキルは **上限なし** のスコアとして扱う。

| スキル           | 概要                   | 算出方法                                                                                                         |
| ---------------- | ---------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **実装力**       | コードを書く技術力全般 | `過去1年の総コントリビューション数` をそのままスコア化                                                             |
| **企画力**       | 言語化・ドキュメント力 | `非Forkリポジトリ数 + Wiki有効数 + 技術の多様性` を合算したスコア                                                 |
| **開発速度**     | アウトプットの速さ     | `直近1年の月次コミットペース + マージ済みPR数` を合算したスコア                                                   |
| **レビュー力**   | チーム改善力           | `PR Review 貢献数 + Issue 貢献数 + マージ済みPR数の一部` を合算したスコア                                         |
| **持続力**       | 継続して進める力       | `最長コントリビューション連続日数 + 年間活動量の一部` を合算したスコア                                             |
| **適応力**       | 技術の広さと順応性     | `使用言語数 + 一定規模以上の使用言語数 + 非Forkリポジトリ数` を合算したスコア                                     |

> **プライベートリポジトリの価値**: パブリックリポジトリのみでは業務コードや個人開発の大部分が反映されないため、プライベートリポジトリを含めることで実際の開発活動をより忠実に反映できる。

### スキル傾向（特化型 vs バランス型）

GitHubのアクティビティパターンを分析し、キャラクターに「特化型」または「バランス型」の傾向を付与する。  
「完璧なキャラクター」が生まれにくくなることで、デッキ構築の戦略性が生まれる。

| 傾向           | 条件                        | 特徴                        |
| -------------- | --------------------------- | --------------------------- |
| **実装特化型** | コミット頻度が高い          | 実装力・開発速度↑ / 企画力↓ |
| **設計特化型** | README / wiki / Issueが充実 | 企画力↑ / 開発速度↓         |
| **バランス型** | 偏りが少ない                | すべてが中程度              |

### 性格診断

初回ログイン時または任意のタイミングで性格診断を実施し、一部スキルへの補正を決定する。

> 現在の実装では性格診断は未実装。

### ランク表示

現在の実装ではキャラクターカードのランクは `tendency` ではなく **デッキスコア** に応じて決まる。
また、星表示はランクの固定値ではなく、**そのランク帯の中でどれだけ上位に近いか** を 1〜5 の5段階で表す。

| ランク名               | 目安デッキスコア | 説明                                                                 |
| ---------------------- | ---------------- | -------------------------------------------------------------------- |
| **トレーニー**         | 0以上            | プログラミング初心者など。                                           |
| **ジュニアエンジニア** | 150以上          | プログラミングにある程度慣れた人                                     |
| **シニアエンジニア**   | 350以上          | プログラミングを仕事にでき、エンジニアになることができる人           |
| **テックリード**       | 700以上          | エンジニアの中でも比較的スキルが高いと位置づけることができる人       |
| **アーキテクト**       | 1200以上         | 特に高いスキルを持ち、CTOやコンサルティングのマネージャーに務まりうる人 |

### 技術特性

技術の判定は **GitHub API の言語統計（Byte数）** をもとに行う。全リポジトリ（パブリック＋プライベート）の言語Byte数を集計し、上位2言語を主要技術として自動設定する。

| 種別         | 最大数                         | 効果                                               |
| ------------ | ------------------------------ | -------------------------------------------------- |
| **主要技術** | 2つまで（上位2言語を自動設定） | 大会との技術マッチ時、全スキルに +3%               |
| **保有技術** | 上限なし（全使用言語が対象）   | 主要技術または保有技術に合致しない大会には出場不可 |

## 大会：ハッカソン

### 概要

大会はターン制で進行し、最終ターンまでに積み上げた「開発スコア」を競う。  
対戦相手は**デッキスコアに近い順**でマッチングされる（非同期対人）。リアルタイム対戦は行わず、相手の登録済みデッキに対してシミュレーションを実行する方式。

### ターン構成

ターン数は難易度によって以下の3段階で固定する。

| 難易度 | ターン数（経過日数） | 企画フェーズ | 実装フェーズ |
| ------ | -------------------- | ------------ | ------------ |
| 初級   | 7ターン              | 前半2ターン  | 残り5ターン  |
| 中級   | 10ターン             | 前半2ターン  | 残り8ターン  |
| 上級   | 14ターン             | 前半2ターン  | 残り12ターン |

**企画フェーズ（前半2ターン）**  
企画力で開発進捗が積み上がる。

**実装フェーズ（残りターン）**  
実装力・開発速度で開発進捗が積み上がる。

### デッキ規制

- デッキに編成できるキャラクターの最大人数は **7人**
- 毎ターン、その時点のフェーズでデッキ全員が行動する

### 勝利条件

```
開発スコア = 実装フェーズの累計開発進捗 × 全メンバーの企画力合計
```

最終ターン終了時点で**開発スコアが高い方が勝利**。  
同点の場合は全メンバーの実装力合計が高い方を勝者とする。

> **設計意図**: 実装フェーズの進捗が勝利条件のベースとなり、企画力が乗数として機能する。
> 実装力・開発速度の高いキャラクターが進捗を稼ぎ、企画力の高いキャラクターがそれを増幅させるため、すべてのスキルが勝利に直結する。

### 毎ターンの補正

各ターンの開始時に、体調・モチベーション・イベントの3種類の補正が独立して抽選される。

#### 体調

| 状態           | 効果          | 発生確率 |
| -------------- | ------------- | -------- |
| すっきり寝れた | 実装力 +10%   | 20%      |
| 普通           | 補正なし      | 70%      |
| 体調不良       | 実装力 -10%   | 9%       |
| 病気           | 全スキル -30% | 1%       |

#### モチベーション

| 状態       | 効果          | 発生確率 |
| ---------- | ------------- | -------- |
| 最高       | 開発速度 +15% | 18%      |
| 普通       | 補正なし      | 80%      |
| やる気なし | 開発速度 -20% | 2%       |

#### アクシデント・ラッキーイベント

毎ターン、デッキ全員の企画力の平均値をもとに抽選される。  
ラッキーイベントとアクシデントは独立して判定する（両方発生しうる）。

| 種別                   | 内容                                  | 発生確率                         |
| ---------------------- | ------------------------------------- | -------------------------------- |
| ラッキー：円滑な開発   | そのターン全員の開発速度 +5%          | 正規化後の企画力に応じて上昇     |
| アクシデント：仕様変更 | そのターンの開発進捗が現時点から -30% | 正規化後の企画力が低いほど上昇   |

> **現在の実装**: 企画力は上限なしスコアのため、そのまま確率に使わず `avgPlanning / (avgPlanning + 200)` で 0〜100 相当に正規化してからイベント確率へ変換している。

> **設計意図**: 企画力が高いチームほど仕様変更リスクが低く、開発が円滑に進みやすくなる。ドキュメントや設計が充実したエンジニアをデッキに組み込む戦略的価値が生まれる。

## マッチングシステム

### 概要

リアルタイム対戦は行わない。パワプロの「栄冠ナイン」や艦これの「演習」と同様に、**他プレイヤーが事前にデータベースへ登録したデッキ**の中から自分のデッキスコアに近いものを取得し、それに対してシミュレーションを実行する非同期方式を採用する。

相手プレイヤーがオンラインである必要はなく、常に対戦が可能。

### デッキ登録

- プレイヤーはいつでも「デッキを登録する」操作で自分の現在のデッキをデータベースに保存できる
- 登録されたデッキはデッキスコアとともにサーバーに保持され、他プレイヤーのマッチング対象となる
- デッキを更新すると登録内容も上書きされる（常に最新の登録デッキが対戦相手として使われる）

### デッキスコア

マッチングの基準となる指標。デッキの総合戦力を表す上限なしスコア。

```
デッキスコア = Σ（各メンバーのスキル加重合計）

スキル加重合計（1人あたり）=
    実装力 × 0.30
  + 開発速度 × 0.23
  + 企画力 × 0.17
  + レビュー力 × 0.12
  + 持続力 × 0.10
  + 適応力 × 0.08

```

> **重み付けの意図**: 勝利条件への直接寄与が大きい実装力・開発速度を高ウェイトにしつつ、長期運用やチーム貢献、技術適応の要素も総合スコアへ反映する。

**参考例（上限なし）**:

| ケース                                                         | デッキスコア（例） |
| -------------------------------------------------------------- | ------------------ |
| 直近活動が少ない初期デッキ                                     | 低め               |
| 継続的に開発しているバランス型デッキ                           | 中程度             |
| 高頻度コミットかつPR運用も活発な実装寄りデッキ                 | 高め               |

## 検索マッチング

### マッチングロジック

1. GitHubのIDで検索して、このゲームに登録されているか確認する
2. 登録されていたら、相手のメインデッキを取得
3. そのデッキと自分の選択したデッキで対戦

### マッチングロジック

1. プレイヤーが大会にエントリーすると、自分の現在のデッキスコアを算出する
2. データベースに登録済みのデッキの中から、**同一難易度帯**かつ**デッキスコアが近い順**に相手候補を取得する（スコア差 ±100 以内を優先）
3. 該当するデッキが見つからない場合は、スコア差の許容範囲を段階的に拡大して再取得する
4. 相手デッキが決定したら、バックエンドがシミュレーションを自動実行する
5. 結果（勝敗・開発スコア詳細・ターンごとの補正ログ）が即時通知される

- 相手プレイヤーのオンライン状態に関わらず即座に対戦可能
- 同一プレイヤーのデッキとの連続マッチングは避ける（直近3戦を除外）
- 相手プレイヤーには「自分のデッキが対戦に使用された」旨が通知される（対戦結果も確認可能）

## 実装メモ

- フロントエンドは Vite + React で動作し、キャラクター生成後は `/base` でステータスカードを表示する
- ステータスカードでは 6スキル、デッキスコア、傾向、デッキスコア由来のランク名、ランク帯内の星評価を表示する
- バックエンドは Express + MySQL 構成で、GitHub OAuth 後に `/auth/me` でアカウント同期を行う
- ローカル開発では `docker compose up --build` で `mysql` と `backend` を起動し、フロントエンドはローカルで `npm run dev` を使う
- DB スキーマはバックエンド起動時に不足列の自動補完を行うため、追加スキル列は既存環境でも段階的に反映される
